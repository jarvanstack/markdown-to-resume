# Preserve Font Colors In Exported Theme CSS

## Metadata

- Status: Completed
- Date: 2026-07-12
- Owner: Codex
- Request: Export theme font colors into CSS so importing the theme preserves those colors instead of showing defaults.
- Related files: `src/lib/customThemes.ts`, `src/lib/storage.test.ts`, `e2e/app.spec.ts`, `docs/knowledge-graph.md`

## Background And Evidence

Theme export currently serializes `ResumeSettings` colors only as import metadata such as `--resume-heading-color` and `--resume-text-color`. Theme rules do not consume those names: built-in and custom theme rules read runtime variables such as `--headerColor`, `--textColor`, `--linkColor`, `--accentColor`, `--accentColorMuted`, and `--mutedColor`.

Inside the editor, `ResumePreview.resumeStyle` normally supplies the runtime variables as inline styles. The exported CSS document itself therefore does not carry directly effective font-color variables and falls back to the color literals embedded in theme rules when used without that inline bridge. This makes exported/imported theme appearance depend on application state rather than the CSS file's own color declarations.

## Goals

- Emit all theme color settings as the runtime CSS variables consumed by theme rules.
- Keep the existing `--resume-*-color` metadata so current import parsing and validation remain compatible.
- Preserve heading, body, link, accent, translucent accent, and muted colors through theme export/import.
- Add focused automated coverage for the serialized CSS contract and the user-visible import workflow.

## Non-Goals

- Change the `ResumeSettings` shape, persisted-state v3, storage keys, or import file format version.
- Move all typography and layout settings from inline preview styles into exported CSS.
- Change built-in theme colors or theme selector scoping.
- Support arbitrary CSS color syntax in import metadata; the existing six-digit hex contract remains.

## Scope And Knowledge-Graph Impact

Affected nodes:

- Custom themes (`src/lib/customThemes.ts`): exported metadata root also owns directly effective runtime color variables.
- Resume rendering (`src/components/ResumePreview.tsx`): no code change; its established runtime variable names define the serialization contract.
- Theme CSS (`src/themes/*.css`): no code change; existing rules consume the newly serialized variables.
- Unit and E2E tests: protect the CSS declaration mapping and import/render behavior.

The existing Theme Catalog -> Custom Themes -> scoped CSS -> Resume Preview edge remains. Its contract becomes stronger: serialized CSS contains both machine-readable import metadata and the runtime color variables consumed by theme rules. No new node or persistence edge is introduced.

## Implementation Plan

1. Extend the `.resume-theme` metadata block generated by `serializeThemeCss` with the six runtime color variables used by resume theme rules.
2. Derive `--accentColorMuted` from the configured six-digit accent color using the same `b3` alpha suffix as `resumeStyle` so exported CSS matches live preview behavior.
3. Keep the `--resume-*-color` declarations unchanged for backward-compatible import parsing.
4. Strengthen unit coverage to assert that configured non-default colors appear under both metadata and runtime variable names.
5. Strengthen the theme download/import E2E workflow to use non-default exported colors and verify the imported theme's computed heading, body, and link colors.
6. Run focused unit and E2E tests, then `npm run check` and `git diff --check`.
7. Update the knowledge graph runtime flow, custom-theme ownership/invariant/test map, and change ledger to describe the final contract.

## Risks, Mitigations, And Rollback

- Risk: Duplicated metadata/runtime declarations could drift. Mitigation: generate both declaration sets together from the same `ResumeSettings` object and assert exact mappings in tests.
- Risk: The translucent accent differs from live preview. Mitigation: use the same `${accentColor}b3` representation already produced by `resumeStyle`.
- Risk: Inline preview variables override file declarations. Mitigation: this is intentional; imported settings are parsed from the same metadata values, while the file remains independently effective when inline variables are absent.
- Rollback: Remove the six runtime declarations and their assertions. No stored data migration or cleanup is required because the file format remains backward compatible.

## Decisions And Deviations

- Decision: Keep semantic `--resume-*-color` metadata and runtime camel-case variables together rather than rename either side, preserving both old imports and all existing theme CSS.
- Decision: Include accent and muted color variables in addition to heading/body/link because borders, bullets, rules, and secondary text are part of the same color contract.
- Deviation: The E2E test uses the stable ordered inputs inside `.color-section` instead of translated accessible labels so the color setup remains valid across Chinese copy changes; user-facing controls and accessibility are unchanged.

## Final Result

Theme CSS serialization now writes heading, body, link, accent, translucent accent, and muted colors under the exact camel-case custom properties consumed by all resume theme rules. The existing `--resume-*-color` declarations remain in the same scoped metadata block for backward-compatible import parsing, and both sets derive from the same `ResumeSettings` values.

The unit contract uses non-default colors to protect every emitted declaration. The browser workflow creates a colored custom theme, downloads it, injects the downloaded file without application inline variables to verify standalone heading/body/link colors, deletes the local source, imports the file again, applies it, and verifies the same computed colors before retaining the existing rename/delete persistence checks.
